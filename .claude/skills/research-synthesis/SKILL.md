# Research Synthesis

**This skill is scaffolded and needs to be designed through a live session — similar to how research-framing was built.** The interaction pattern should be developed by walking through real post-study synthesis work.

## Intended Purpose

Help the user sharpen insight statements, validate patterns, check for researcher bias, and ensure evidence is weighted appropriately — all without ever receiving raw user data. This skill operates during Hall's Analysis and Synthesis stage (after data collection).

## Nikki Anderson Prompts to Integrate

Detailed prompt content should be added to references/nikki-synthesis-prompts.md. The following prompts from Nikki's article map to this stage:

- **Insight Clarity:** For sharpening wording, scanning for ambiguity and bias, assessing evidence strength
- **Pattern Validation:** For stress-testing patterns with alternative interpretations, rival hypotheses, and blind spot identification
- **Decision-Ready Framing:** For translating a validated insight into decision-focused language (headline, business impact, recommendation, risk of inaction)
- **Bias + Blind Spot Detector:** For surfacing confirmation bias, pattern bias, recency bias, and missing perspectives

## Key Calabro Application

At this stage, the critical question is: "Is a Category C finding (a user quote, a sentiment rating) being used to override Category A evidence (observed task failure, behavioral patterns)?" This is where teams most commonly give weak evidence too much weight because it's easier to package.

## Key Guardrail

The user provides only their own summaries, interpreted patterns, and drafted insight statements — never raw data, transcripts, quotes, or identifiable participant information. If the user starts pasting what looks like raw data, stop and remind them of this boundary.

## Design Notes for Future Build

- The reflect-back pattern here should focus on restating the user's insight in clearer form and asking "is this what you mean?"
- The skill should proactively ask about evidence that *contradicts* the pattern, not just evidence that supports it
- Alternative interpretations should always be offered, even when the user seems confident
- The distinction between "what I observed" and "what I think it means" should be maintained throughout
